# 🏗️ هيكل المشروع المقترح — التنظيم حسب الميزة

> الوثيقة التشغيليّة لإعادة تنظيم **شجرة الملفّات** في مستودع الكود من تقسيمٍ بالطبقة التقنيّة المتناثرة (مفسّر/مترجم/أدوات) إلى تقسيمٍ **حسب الميزة** — فتجتمع شظايا كلّ ميزة في مجلّد واحد. هذه عدسة **نظام الملفّات**، مكمِّلة لـRFC sadlang-rfcs#10 الذي ثبّت **حدود أهداف CMake** (طبقة الربط لا طبقة المجلّدات).

> ⚠️ **هذا اقتراحٌ موثَّق لا تنفيذ.** أيّ نقلٍ فعليّ للمجلّدات تغييرٌ بنيويّ كاسرٌ محتمل ⇒ يمرّ عبر **RFC تنفيذ مخصّص + PRs مرحليّة غير كاسرة**، مع إبقاء حارسَي الحدود مارَّين في كلّ خطوة.

---

## 1. لماذا؟ — المشكلة الراهنة

شجرة الجذر اليوم مقسّمة **بالطبقة التقنيّة** (`shared/` · `interpreter/` · `compiler/` · `tools/` · `stdlib/` …)، فتتناثر **الميزة الواحدة** عبر مجلّدات كثيرة. مثال **الرسومات/الواجهات** يتوزّع على **تسعة مواقع**:

| الشظيّة | الموقع الحاليّ |
|---|---|
| محرّك الواجهات | `sad_ui/` (core/backends/pipeline/reactive/widgets) |
| تحليل الواجهات | `shared/parser/src/ui/parser_ui.cpp` |
| جسر المفسّر | `interpreter/src/ui/` + `interpreter/include/ui/` |
| باعث المترجم | `compiler/src/ui/` (بذرة `sad_ui_ir` الميتة) |
| وقت التشغيل | `runtime/sad_ui_runtime.{h,cpp}` |
| مكتبة قياسيّة | `stdlib/رسومات.ص` + `stdlib/image/` + `stdlib/audio3d/` |
| مكتبة الطرف الثالث | `graphics/third_party/SDL2/` |
| مضيف الجوال | `platform/android/` |
| مصدر الحقيقة | `language-truth/builtins/{ui_widgets,ui_platform,compiler_ui}.yaml` |

**الأثر:** فهمُ ميزةٍ أو تعديلُها يتطلّب القفز بين تسعة مجلّدات؛ والمساهم الجديد لا يرى «أين الرسومات». ويتفاقم ذلك بمشكلتين جانبيّتين:
- **ازدواج:** `data/` يتقاطع مع `language-truth/` (مصدر الحقيقة)؛ `وثائق/` + `specs/` يتقاطعان مع `docs/`.
- **ضوضاء متتبَّعة في الجذر:** `_bmad/` · `_bmad-output/` · `ملفات_التدريب/` · `_proj_pdf/` · `_scratch/` · `تقارير_مؤقته/` · `templates/` — كلّها داخل المستودع وتشوّش الجذر.

---

## 2. المبدأ الحاكم

> **`core/` = ما يشترك فيه كلّ شيء؛ `features/` = كلّ ميزة تجمع شظاياها عبر الطبقات؛ الباقي = أدوات/توثيق/بناء.**

ثلاثة أنواع للمجلّدات في الجذر:
1. **النواة والمحرّكان (`core/`):** ما لا ينتمي لميزةٍ بعينها — اللغة نفسها (معجم/محلّل/أنواع) والمحرّكان.
2. **الميزات (`features/`):** كلّ قدرةٍ يراها المستخدم (رسومات، شبكة، تعمية …) تجمع شظاياها (تحليل + تفسير + ترجمة + وقت تشغيل + مكتبة + بيانات) في مجلّد واحد.
3. **المحيط:** أدوات، توثيق، بناء، اختبارات، توزيع — مجلّدات أفقيّة تخدم الكلّ.

---

## 3. ⚠️ التوافق الحرج مع RFC sadlang-rfcs#10 (الحدود لا تنكسر)

التقسيم حسب الميزة **عدسة تنظيميّة لا تُرخي حدود المحرّكَين**. فحتّى لو اجتمعت `features/ui/interp/` و`features/ui/compile/` في مجلّدٍ أبٍ واحد:
- يجب أن يُجمَّع `features/ui/interp/*` في هدف `sad_ui_bridge` (ضمن **نظام المفسّر**)، و`features/ui/compile/*` في هدف `sad_compiler` (ضمن **نظام المترجم**) — **بلا روابط متقاطعة**.
- يبقى حارسا `check_layering.py` (طبقات الربط) و`check_interpreter_boundary.py` (حدّ التضمين) **مصدرَ الفرض**؛ المجلّدات راحةٌ بصريّة، لا حوكمة.

> القاعدة: **بنية المجلّدات حسب الميزة، وبنية أهداف CMake حسب المحرّك — وكلاهما مفروض.** أيّ نقل يجب أن يُبقي الخريطة `مجلّد ميزة ⟶ هدف صحيح` سليمة، ويُتحقَّق منها بالحارسَين في كلّ PR.

## 4. الهيكل المقترح (الشجرة الهدف)

```
s-programming-language/
├── x.py · CMakeLists.txt           # المنسّق + جذر البناء
│
├── core/                           # ① النواة والمحرّكان (طبقات RFC sadlang-rfcs#10)
│   ├── shared/                     #   sad_shared — نواة اللغة (معجم/محلّل/AST/أنواع/قيمة/أخطاء)
│   ├── interp/                     #   sad_interp — المفسّر الشجريّ + sad_runtime
│   └── compile/                    #   sad_compiler — مولّد LLVM
│
├── features/                       # ② الميزات — مجلّد كامل لكلّ ميزة عبر الطبقات
│   ├── ui/                         #   الرسومات/الواجهات (يجمع المواقع التسعة)
│   │   ├── engine/                 #     ← sad_ui/ (core/backends/…)            ⟶ هدف sad_ui
│   │   ├── parser/                 #     ← shared/parser/src/ui/
│   │   ├── interp/                 #     ← interpreter/src/ui + include/ui  ⟶ هدف sad_ui_bridge
│   │   ├── compile/                #     ← compiler/src/ui (sad_ui_ir)       ⟶ هدف sad_compiler
│   │   ├── runtime/                #     ← runtime/sad_ui_runtime.*
│   │   ├── stdlib/                 #     ← stdlib/{رسومات.ص,image,audio3d}
│   │   └── vendor/                 #     ← graphics/third_party/SDL2
│   ├── network/                    #   ← network/ + stdlib/network + tools/wasm? + شبكات.ص
│   ├── crypto/                     #   ← stdlib/crypto (+ سدّ تباعد المترجم DIVERGENT(crypto))
│   ├── concurrency/                #   ← stdlib/async + التزامن
│   ├── data/                       #   ← stdlib/{json,xml,database,خرائط.ص}
│   ├── filesystem/ · io/ · math/   #   ← stdlib/{filesystem,io,math,رياضيات.ص}
│   ├── lowlevel/                   #   ← stdlib/low_level + freestanding + embedded
│   └── mobile/                     #   ← platform/android + tools/apk_builder
│
├── stdlib/                         # ③ مكتبة قياسيّة عامّة (.ص + C++) غير مرتبطة بميزةٍ بعينها
├── apps/                           # sad-run · sad-build (رفيعان)
├── tools/                          # lsp · formatter · repl · analyze · sadinfo · … (أوراق)
├── language-truth/                 # مصدر الحقيقة الوحيد (يبتلع data/)
├── docs/                           # توثيق موحَّد (يبتلع وثائق/ + specs/)
├── tests/ · cmake/ · scripts/ · examples/ · third_party/ · distribution/ · deployment/
│
└── (خارج المستودع أو .gitignore):
    _bmad/ · _bmad-output/ · ملفات_التدريب/ · _proj_pdf/ · _scratch/ · تقارير_مؤقته/ · templates/
```

> **ملاحظة `language-truth/`:** يبقى **مركزيًّا** (مصدر حقيقة واحد للّغة كلّها)، لا يُوزَّع على الميزات — فمنه يُولَّد ما يشترك فيه المحرّكان. يجوز لاحقًا تنظيمه داخليًّا بمجلّدات ميزات، لكنّه يبقى جذرًا واحدًا.

## 5. مصير كلّ مجلّد جذر حاليّ

| المجلّد الحاليّ | المصير المقترح |
|---|---|
| `shared/` | ⟶ `core/shared/` |
| `interpreter/` | ⟶ `core/interp/` |
| `compiler/` | ⟶ `core/compile/` |
| `sad_ui/` | ⟶ `features/ui/engine/` |
| `graphics/` | ⟶ `features/ui/vendor/` (SDL2 طرف ثالث) |
| `network/` | ⟶ `features/network/` |
| `runtime/` | ⟶ يُقسَّم: `sad_ui_runtime` ⟶ `features/ui/runtime/`؛ المضمَّن ⟶ `core/compile/runtime/` |
| `platform/` | ⟶ `features/mobile/` |
| `stdlib/` | ⟶ يُقسَّم: الميزات المحدّدة ⟶ `features/*/stdlib/`؛ العامّ يبقى `stdlib/` |
| `data/` | ⟶ يُدمَج في `language-truth/` (ازدواج §ز) ثمّ يُحذف |
| `وثائق/` + `specs/` | ⟶ يُدمَجان في `docs/` |
| `apps/` · `tools/` · `language-truth/` · `docs/` · `tests/` · `cmake/` · `scripts/` · `examples/` · `third_party/` · `distribution/` · `deployment/` | تبقى أفقيّة (تخدم الكلّ) |
| `_bmad/` · `_bmad-output/` · `ملفات_التدريب/` · `_proj_pdf/` · `_scratch/` · `تقارير_مؤقته/` · `templates/` | تُنقَل خارج المستودع أو تُضاف لـ`.gitignore` (ضوضاء/أدوات وكلاء/مؤقّت) |

## 6. خطة الهجرة (مرحليّة، غير كاسرة، عبر RFC)

كلّ خطوة = PR واحد يُبقي البناء أخضر والحارسَين مارَّين:

1. **هـ0 — تجميد القرار:** RFC تنفيذ مخصّص يثبّت الشجرة الهدف وخريطة `مجلّد ⟶ هدف`.
2. **هـ1 — تنظيف المحيط (الأقلّ خطرًا):** نقل/إخراج الضوضاء (`_bmad*` · `ملفات_التدريب` · `_proj_pdf` · `_scratch` · `تقارير_مؤقته` · `templates`) + دمج التوثيق (`وثائق/` + `specs/` ⟶ `docs/`).
3. **هـ2 — إزالة الازدواج:** دمج `data/` في `language-truth/` (بعد إثبات أنّ المولّدات تقرأ المصدر الموحَّد).
4. **هـ3 — ميزة البرهان (الرسومات):** نقل الشظايا التسعة إلى `features/ui/*` مع تحديث مسارات CMake فقط (الأهداف نفسها لا تتغيّر) — **دليل المفهوم**.
5. **هـ4 — بقيّة الميزات:** `network` · `crypto` · `data` · `mobile` … واحدةً تلو الأخرى.
6. **هـ5 — `core/`:** نقل `shared`/`interpreter`/`compiler` إلى `core/` (الأوسع أثرًا، أخيرًا).

> **معيار كلّ خطوة:** `git mv` يحافظ على التاريخ؛ تُحدَّث مسارات CMake فقط؛ بناء `BUILD_TESTS=ON` كامل + الحارسان + البوّابة المزدوجة بلا تراجع؛ مراجعة قبل الدفع.

---

## 7. المكسب المتوقَّع

- **اكتشافيّة:** «أين الرسومات؟» ⟵ `features/ui/` (مجلّد واحد بدل تسعة).
- **عزل التغيير:** تعديل ميزة يلامس مجلّدها وحده غالبًا.
- **حدود أوضح للمساهمة:** صلاحيات/ملكيّة لكلّ ميزة (CODEOWNERS لكلّ `features/*`).
- **تنظيف الجذر:** اختفاء الضوضاء والازدواج.
- **بقاء الفرض:** حدود المحرّكَين تبقى مفروضة بالحارسَين (المجلّدات لا تُرخيها).
