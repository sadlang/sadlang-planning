# 🧩 unified-core-rfc10 — القلب الموحَّد بحدود داخلية

> إعادة تنظيم قلب لغة **ص** إلى مساحة عمل واحدة (monorepo) بحدود وحدات داخلية **مفروضة حاسوبيًّا** — بدل تشظية المستودعات — فيبقى المفسّر والمترجم متوافقَين سلوكيًّا تحت سقفٍ واحد، وتُصان الحدود بالأدوات لا بالنيّة.

| الحقل | القيمة |
|---|---|
| **الحالة** | `قيد التنفيذ` (م0–م5 مُنجزة؛ تأجيلان قائمان) |
| **المالك** | فريق القلب (core) |
| **مستودع الكود** | https://github.com/sadlang/s-programming-language |
| **مرجع التصميم** | RFC sadlang-rfcs#10 (قلب موحَّد بحدود داخلية) |
| **تاريخ البدء** | 2026-06-25 |
| **آخر تحديث** | 2026-06-28 |

## 🎯 الهدف

جعل قلب اللغة **يبدو ويعمل كلغةٍ عالميّة المستوى** (مثل Rust/Python/Go) عبر **مساحة عمل واحدة بحدود داخلية صارمة** — لا فصل مستودعات. الحدود موجودة فعلًا لكنّها كانت **غير موثَّقة وغير مفروضة**، فينشأ سوء فهم متكرّر («`sad_core` هو نواة اللغة») وتسرّب عَرَضيّ صامت. يحلّ المشروع ذلك بـ:

- **نواة لغة حقيقيّة واحدة** (`sad_shared`: معجم/محلّل/AST/أنواع/قيمة/أخطاء + المصدر المولَّد من مصدر الحقيقة) يبني عليها المحرّكان.
- **عزل المحرّكَين** (المفسّر الشجريّ `sad_interp` ومترجم LLVM `sad_compiler`) خلف حدود CMake، بلا روابط مباشرة بينهما.
- **فرض الحدود آليًّا** بحارسَين في CI (طبقات الربط + حدّ تضمين المفسّر) لا بالمراجعة اليدويّة.
- **منسّق بناء ذرّيّ** (`x.py`) يبني المحرّكَين في التهيئة نفسها، ومصدر حقيقة واحد (`language-truth/*.yaml`) يُولّد ما يشترك فيه المحرّكان.

المستفيد: مساهمٌ يفهم البنية فورًا بلا التباس، وفريق اللغة الذي يريد محرّكَين بسلوكٍ متطابق وحدودٍ لا تنحدر بصمت.
