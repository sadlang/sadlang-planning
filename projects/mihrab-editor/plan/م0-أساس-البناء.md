# م0 — أساس البناء (تفصيل تنفيذيّ)

> الهدف: إنتاج ثنائيّ محراب يَبني محلّيًّا من منبع VSCodium دون أيّ تعديل وظيفيّ بعد، للتحقّق من خطّ البناء قبل أيّ تخصيص. **لا نبني فوق أساس لا نثق ببنائه.**

## لماذا VSCodium لا microsoft/vscode مباشرة

| الجانب | `microsoft/vscode` | `VSCodium/vscodium` |
|---|---|---|
| إزالة التتبّع والعلامة الملكيّة | يدويّ، يتكرّر كل إصدار | **جاهز** (رُقَع تُطبَّق آليًّا) |
| بنية الرُقَع (`patches/`) | لا توجد | **موجودة** ومُؤتمتة |
| سوق افتراضيّ | سوق مايكروسوفت (ممنوع للتفرّع) | **Open VSX** مضبوط مسبقًا |
| خطّ بناء متعدّد المنصّات | تبنيه بنفسك | **سكربتات `build/`** جاهزة |

⇒ VSCodium يختصر طبقة كاملة من الطبقة-3 (الرُقَع) ويهديك إيّاها مصونة.

## المتطلّبات المسبقة (بيئة البناء)

- **Node.js** بالإصدار الذي يثبّته `.nvmrc` في منبع VSCodium (لا الأحدث — طابقه حرفيًّا).
- **Python 3** (لـ`node-gyp`).
- أدوات بناء أصليّة:
  - Windows: «Visual Studio Build Tools» (C++ workload) + Windows SDK.
  - (لاحقًا) macOS: Xcode CLT؛ Linux: build-essential.
- مدير الحزم الذي يحدّده المنبع (Yarn classic عادةً).
- مساحة قرص ≥ 30GB ووقت بناء أوّل طويل (Electron + Chromium).

## الخطوات

1. **الاستنساخ**
   - استنسخ `VSCodium/vscodium` (لا تتفرّعه على GitHub بعد — استنساخ محلّيّ للإثبات أولًا).
   - ثبّت عند **وسم إصدار مستقرّ** (tag) لا على `master`، لتثبيت قاعدة معروفة.

2. **التحقّق من البيئة**
   - طابق إصدار Node مع `.nvmrc`.
   - شغّل تثبيت التبعيّات النظيف.

3. **بناء نظيف دون تخصيص**
   - شغّل سلسلة بناء VSCodium القياسيّة كما هي (تطبّق رُقَعها على منبع vscode المسحوب).
   - الهدف: ثنائيّ VSCodium يعمل، لإثبات أنّ البيئة سليمة.

4. **بوّابة الخروج (Definition of Done لـم0)**
   - الثنائيّ يُبنى ويُشغَّل على Windows (المنصّة الأساسيّة).
   - وثّق: إصدار Node، إصدار VSCodium tag، زمن البناء، أيّ عقبات بيئيّة، في `reports/`.

## المخاطر الخاصّة بـم0

- **عدم تطابق إصدار Node** ⇒ فشل `node-gyp`. علاجه: التزام `.nvmrc` حرفيًّا.
- **غياب أدوات بناء C++ الأصليّة** على Windows ⇒ فشل تجميع وحدات أصليّة. علاجه: تثبيت Build Tools قبل البدء.
- **البناء من `master` المتحرّك** ⇒ نتائج غير قابلة لإعادة الإنتاج. علاجه: التثبيت على tag.

## ما يلي مباشرةً (مدخلات م1 وم2 — للتخطيط لا التنفيذ الآن)

### م1 — الهويّة (نقاط التلامس الملموسة)
- `product.json`: الحقول الدنيا التي تُغيَّر — `nameShort` و`nameLong` («محراب»)، `applicationName`، `dataFolderName`، `win32` معرّفات التثبيت، `urlProtocol`، `extensionsGallery` (Open VSX).
- أصول العلامة: الأيقونات (ico/icns/png بأحجامها)، شاشة الترحيب. تُحقَن بنمط رُقَع VSCodium في `src/stable/` أو ما يماثله.
- **لا منطق نواة في م1 إطلاقًا** — كله بيانات وأصول.

### م2 — حزم سلسلة أدوات ص (أكبر قطعة قيمة)
ثلاثة أصول تُحزَم داخل المنتج بتهيئة صفريّة:
1. **إضافة ص الناضجة** — تُوضَع كـbuiltin extension (مجلّد `extensions/` المدمج) لا من السوق.
2. **خادم LSP** — يُرفَق كأصل ثنائيّ، وتشير الإضافة إليه بمسار نسبيّ داخل التطبيق.
3. **المترجم/المشغّل** (`sad-build` / `sad-run`) — يُرفَقان كأصول، مع ضبط PATH داخليّ.
- **جرد مطلوب قبل م2**: فحص الإضافة الحاليّة لتحديد (أ) ما يُحزَم كما هو، (ب) ما يحتاج رُقعة نواة (RTL ⇒ يُؤجَّل لـم3)، (ج) أيّ أسرار/مفاتيح يجب ألّا تدخل منتجًا عامًّا.
