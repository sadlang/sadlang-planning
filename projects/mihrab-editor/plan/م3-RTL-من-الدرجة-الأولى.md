# م3 — RTL من الدرجة الأولى (تفصيل تنفيذيّ)

> هذه المرحلة هي **المبرّر الجوهريّ الوحيد** لبناء محرّر خاصّ بدل الاكتفاء بإضافة. VS Code الأصليّ لا يدعم تخطيط واجهة RTL، ودعمه للنصّ ثنائيّ الاتّجاه (bidi) مضبوط للّغات اللاتينيّة. لغة ص بكلماتها ومعرّفاتها العربيّة تحتاج تجربة RTL أصيلة لا التفافًا.
>
> ⚠️ هي أيضًا **أعلى مراحل دَيْن الدمج** (الطبقة 3). كل رُقعة نواة هنا نقطة احتكاك دائمة مع المنبع. المبدأ الحاكم: **ادفع كل ما يمكن إلى طبقة الإضافة؛ ولا تنزل إلى النواة إلّا لما يستحيل فوقها — وبأصغر رُقعة ممكنة.**

## تصنيف الحدّ: الإضافة مقابل النواة

### ✅ يُحَلّ في طبقة الإضافة (دَيْن دمج صفر)
يبني على مشروع [vscode-rtl](../../vscode-rtl/) القائم (نمط Strategy، نقطة تسجيل `buildRegistry()`):

- **إعداد اللغة** (`language-configuration`): الأقواس، الإغلاق التلقائيّ، التعليقات — محايد اتّجاهيًّا لكن يضبط سلوك التحرير.
- **التلوين الدلاليّ** (semantic highlighting) للكلمات والأنواع العربيّة.
- **زخارف المحرّر** (decorations) القابلة للحقن عبر واجهة الإضافات.
- **أوامر وإعدادات** لتبديل سلوكيّات اتّجاهيّة لكل مستند.
- **اكتشاف الاتّجاه لكل سطر** عبر منطق الإضافة حيث تسمح الواجهة.

### 🔧 يفرض رُقعة نواة (الطبقة 3 — احصرها ووثّقها)

كل بند أدناه مستحيل في طبقة الإضافة لأنه يمسّ DOM/CSS النواة أو محرّك Monaco:

| البند | لماذا لا يكفي الإضافة | حجم الرُقعة المتوقّع |
|---|---|---|
| **تخطيط واجهة workbench بـ`dir="rtl"`** | شريط النشاط/الجانبيّ/اللوحات مثبّتة LTR في DOM/CSS النواة؛ لا واجهة إضافة تقلبها | متوسّط–كبير (CSS + تخطيط) |
| **اتّجاه محرّر Monaco الأساسيّ** | اتّجاه النصّ وحركة المؤشّر والتحديد في محرّك التحرير لا تُكشَف للإضافات | كبير (قلب المحرّر) |
| **معالجة محارف bidi التحكّميّة** | افتراض VS Code (تمييز Trojan Source) عدائيّ للعربيّة-كشيفرة؛ يحتاج ضبطًا في النواة | صغير–متوسّط |
| **التفاف الأسطر وكشف الاتّجاه التلقائيّ في المحرّك** | منطق word-wrap واتّجاه الفقرة داخل Monaco | متوسّط |

## استراتيجيّة احتواء دَيْن الدمج

1. **تدرّج لا قفزة**: ابدأ بأصغر رُقعة عالية القيمة (معالجة محارف bidi للعربيّة) — أثر كبير، رُقعة صغيرة. أجّل تخطيط workbench الكامل إن أمكن.
2. **رُقَع ذرّيّة موثّقة**: كل رُقعة في ملفّ مستقلّ تحت `patches/`، مع تعليق رأسيّ يشرح: ماذا، لماذا، وأيّ ملفّ منبع تمسّ.
3. **تفضيل الحقن على التعديل**: حيثما أمكن، أضِف سلوكًا (CSS class، إعداد) بدل تعديل منطق قائم — أسهل في إعادة التطبيق.
4. **اختبار إعادة التطبيق في CI** (م5): أيّ رُقعة تفشل في الانطباق على إصدار منبع جديد تُرفَع كإنذار فوريّ.
5. **عدّاد رُقَع النواة = مؤشّر الصحّة**: راقبه؛ نموّه المطّرد إنذار انجراف نحو نمط التفرّع العميق المرفوض.

## ترتيب التنفيذ المقترح داخل م3

1. **استيعاب ما تنجزه إضافة vscode-rtl** (الطبقة 1) أولًا — لا تكرّره في النواة.
2. **رُقعة محارف bidi** (صغيرة، أثر عالٍ) — أوّل رُقعة نواة.
3. **اتّجاه محرّر Monaco** للنصّ العربيّ-كشيفرة.
4. **تخطيط workbench RTL** (الأكبر) — يُقيَّم: هل قيمته تبرّر دَيْنه؟ قد يُجزَّأ أو يُؤجَّل.

## بوّابة الخروج (DoD لـم3)

- تجربة تحرير RTL لشيفرة ص لا يضاهيها VS Code الأصليّ.
- كل رُقعة نواة موثّقة وذرّيّة وتمرّ في اختبار إعادة التطبيق.
- عدّاد رُقَع النواة موثَّق ومراقَب.

## المخاطر الخاصّة بـم3

- **رُقعة workbench RTL تتضخّم** ⇒ أعلى مصدر لدَيْن الدمج. علاجه: تجزئة + تقييم قيمة/كلفة قبل الالتزام بها كاملة.
- **تعارض مع تحديثات Monaco في المنبع** ⇒ كسر متكرّر. علاجه: أصغر رُقعة ممكنة + اختبار م5.
- **ازدواج العمل مع إضافة vscode-rtl** ⇒ منطق مكرّر في طبقتين. علاجه: استيعاب حدود الإضافة أولًا (الخطوة 1).
