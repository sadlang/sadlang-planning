# خطّة اختبارات محراب والنظام المتّبع

> محراب ليس برنامجًا واحدًا بل **تركيب**: منبع VSCodium + رُقَع + إضافة ص محزومة + خادم LSP + المترجم/المشغّل + مُثبِّت موقَّع. لذا الاختبار **طبقيّ** يطابق طبقات البناء، وكل طبقة لها بوّابة CI خاصّة. الفلسفة: نختبر كل الحالات ونُبلغ عن الفاشل لا نخفيه بعامل التفاف.

## هرم الاختبار (من الأرخص للأغلى)

```
            ┌─ E2E محرّر (Electron) ─┐   ← قليلة، بطيئة، حرجة
         ┌──┴─ تكامل (LSP/المترجم) ──┴──┐
      ┌──┴─ وحدة (الإضافة + الرُقَع) ────┴──┐
   ┌──┴─ ثابتة (انطباق الرُقَع + فحص الأسرار) ──┴──┐
```

> الهرم مرتّب بالكلفة لا بالرقم: الفحوص الساكنة الرخيصة (انطباق الرُقَع + مسح الأسرار) في القاعدة، وقمّته اختبارات E2E الغالية. أمّا «الطبقات السبع» أدناه فمرتّبة بالاهتمام لا بالكلفة، فبعض بنود طبقة الأمن (مثل تثبيت/تحديث دخانيّ) تقع أعلى الهرم رغم رقمها الأخير.

## الطبقات السبع

### 1. اختبارات انطباق الرُقَع (الأساس، ترتبط بـم5)
- **ماذا**: هل تنطبق كل رُقعة طبقة-3 نظيفةً على وسم المنبع الحاليّ؟
- **النظام**: CI يسحب وسم VSCodium، يطبّق الرُقَع، يفشل فورًا عند أوّل رفض (reject).
- **البوّابة**: لا دمج لأيّ رُقعة لا تنطبق. عدّاد الرُقَع الفاشلة = إنذار انجراف.

### 2. اختبارات البناء الدخانيّة (smoke)
- **ماذا**: هل يُبنى ثنائيّ محراب المعلَّم ويُقلع على كل منصّة؟
- **النظام**: مصفوفة CI (Windows أولًا، ثم macOS/Linux)؛ إقلاع headless + لقطة الإصدار.
- **البوّابة**: فشل البناء أو الإقلاع على أيّ منصّة = حجب الإصدار.

### 3. اختبارات وحدة الإضافة
- **ماذا**: منطق إضافة ص واستراتيجيّات RTL (يبني على 17 اختبار vscode-rtl القائمة).
- **النظام**: مشغّل اختبار الإضافة القياسيّ، يعمل دون إقلاع المحرّر الكامل.

### 4. اختبارات تكامل LSP والمترجم
- **ماذا**: إكمال، hover، تشخيصات، انتقال للتعريف — ومطابقتها لمصدر الحقيقة (SoT) للغة.
- **النظام**: عميل LSP اختباريّ يخاطب الخادم المحزوم؛ ومجموعة برامج ص ذهبيّة تُترجَم وتُشغَّل عبر `sad-build`/`sad-run` المحزومين بتهيئة صفريّة.
- **البوّابة**: أيّ انحراف عن SoT (راجع تقارير توافق LSP) = فشل لا تحذير.

### 5. اختبارات E2E للمحرّر (Electron)
- **ماذا**: سيناريو مستخدم حقيقيّ — فتح ملفّ `.ص`، كتابة، استدعاء إكمال، رؤية تشخيص، حفظ.
- **النظام**: مشغّل اختبار Electron القياسيّ لـVS Code (`@vscode/test-electron`) + قيادة آليّة للواجهة (Playwright).
- **البوّابة**: قليلة ومركّزة على المسارات الحرجة (بطيئة فلا تُكثَر).

### 6. اختبارات RTL (التمايز — الأصعب)
- **ماذا**: حركة المؤشّر والتحديد في نصّ عربيّ-كشيفرة، رسم bidi، تخطيط workbench عند `dir=rtl`.
- **النظام**:
  - **سلوكيّة**: تأكيد مواضع المؤشّر/التحديد برمجيًّا عبر واجهة المحرّر.
  - **انحدار بصريّ** (visual regression): لقطات مرجعيّة لرسم RTL، مقارنة آليّة لكشف الكسر.
- **البوّابة**: أيّ كسر بصريّ في RTL = فشل (هذا جوهر منتجنا).

### 7. اختبارات الأمن والتعبئة
- **ماذا**:
  - **فحص أسرار**: منع تسرّب أيّ مفاتيح TLS/JWT أو أصول مغلقة إلى بناء عامّ (راجع تحذير حوكمة README).
  - **حداثة Electron/Chromium**: التحقّق ألّا يحمل البناء إصدارًا بثغرات معروفة (يربط م5).
  - **المُثبِّت والتوقيع والتحديث التلقائيّ**: المُثبِّت يعمل، موقَّع، وقناة التحديث تسلّم.
- **النظام**: ماسح أسرار في CI + تدقيق إصدار Electron + اختبار تثبيت/تحديث دخانيّ.
- **البوّابة**: أيّ سرّ مكشوف أو إصدار Electron مثغور = حجب فوريّ.

## النظام الإجماليّ

1. **بوّابات تصاعديّة**: لا تصل تغييرة إلى الطبقة الأغلى قبل اجتياز الأرخص.
2. **CI دوريّ على المنبع** (م5): يعيد تطبيق الرُقَع على آخر وسم VSCodium ويشغّل الطبقات 1–4 ليكشف الانجراف مبكّرًا.
3. **منع التراجع**: كل عطب مُصلَح يُثبَّت باختبار يحرسه (لا إصلاح بلا اختبار حارس).
4. **التقارير في `reports/`**: نتائج كل جولة، مع تمييز الفاشل صراحةً لا إخفائه.
